# Job Search Tracker Application — Product and Engineering Specification

## 1. Goal

Build a local job-search tracking application that periodically ingests jobs by directly scraping data from preconfigured job-search URLs, stores normalized job records in a local database, and provides a user interface for browsing, searching, commenting, favoriting, and marking jobs as applied.

The system must be modular so new job-source parsers can be added cleanly, for example SEEK, LinkedIn, Indeed, company career pages, or other job listing websites.

The implementation must follow maintainable engineering practices: clear boundaries, typed models, idempotent ingestion, database migrations, testable parser contracts, structured logging, and robust scraping logic.

---

## 2. Search Requirement


Each source adapter must declare one of the following ingestion modes:

1. `html_scraper`
2. `manual_import`
3. `disabled`

`html_scraper` sources scrape a search results URL and then parse matching job detail pages.
`manual_import` sources support a manual “paste job URL” workflow. In this mode, the user provides a single job detail URL through the UI, and the system fetches and parses the job page directly rather than discovering links from a search listing.
`disabled` sources are skipped entirely.

All ingestion must be performed by directly scraping publicly accessible HTML pages from the configured URLs or manually-entered job URLs. The system must not rely on official APIs. All job data must be extracted from HTML responses using parsers.

## 3. Core Features

### 3.1 Job Source Configuration

The user can define one or more job search sources.

Each source contains:

```yaml
sources:
  - name: "SEEK AI Jobs NZ"
    provider: "seek"
    description: "SEEK AI and machine learning job listings for New Zealand"
    enabled: true
    ingestion_mode: "html_scraper"
    search_url: "https://nz.seek.com/artificial-intelligence-jobs-in-information-communication-technology?sortmode=ListedDate"
    schedule_minutes: 60

  - name: "SEEK ICT Jobs NZ"
    provider: "seek"
    description: "SEEK ICT job listings for New Zealand"
    enabled: true
    ingestion_mode: "html_scraper"
    search_url: "https://nz.seek.com/jobs-in-information-communication-technology"
    schedule_minutes: 60
```

The backend must load sources from a DB-backed configuration stored in SQLite. A configuration file may seed initial sources, but the authoritative source list and runtime source state are persisted in the database.

Minimum source fields:

| Field              | Type        | Required | Description                                    |
| ------------------ | ----------- | -------- | ---------------------------------------------- |
| `id`               | UUID/string | Yes      | Internal source identifier                     |
| `name`             | string      | Yes      | Human-readable source name                     |
| `provider`         | string      | Yes      | Parser/provider key                            |
| `description`      | string/null | No       | Optional notes or source description           |
| `enabled`          | boolean     | Yes      | Whether scheduled ingestion runs               |
| `ingestion_mode`   | enum        | Yes      | `html_scraper`, `manual_import`, or `disabled` |
| `search_url`       | string/null | Conditional | Preconfigured search URL. Required for `html_scraper`; must be null or ignored for `manual_import` where the user pastes a specific job URL. |
| `schedule_minutes` | integer     | No       | Source-specific polling interval               |
| `last_run_at`      | datetime    | No       | Last ingestion attempt                         |
| `last_success_at`  | datetime    | No       | Last successful ingestion                      |
| `created_at`       | datetime    | Yes      | Creation timestamp                             |
| `updated_at`       | datetime    | Yes      | Update timestamp                               |

| Field             | Type          | Required | Notes                                                  |
| ----------------- | ------------- | -------- | ------------------------------------------------------ |

| `source_id`       | string        | Yes      | Source that discovered the job                         |

| `canonical_url`   | string        | Yes      | Normalized URL used for deduplication and matching     |
| `external_job_id` | string/null   | No       | Provider-specific job ID if available                  |
| `city_location`   | string/null   | No       | City, region, remote, hybrid, etc.                     |

| `employer_name`   | string/null   | No       | Employer or advertiser                                 |
| `employment_type` | string/null   | No       | Full-time, part-time, contract, casual, etc.           |

| `user_comments` | text/null     | null    | Free-text notes                                             |
| `status`        | enum          | `new`   | `new`, `viewed`, `applied`, `rejected`, `archived`, `stale` |

## 5. Database Design

Use a local database. Recommended default: SQLite for single-user local deployment. Use PostgreSQL-compatible schema conventions so the app can migrate later.

Recommended ORM: SQLAlchemy or equivalent.

### 5.1 Tables

#### `job_sources`

```sql
CREATE TABLE job_sources (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    provider TEXT NOT NULL,
    description TEXT,
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    ingestion_mode TEXT NOT NULL,
    search_url TEXT,
    schedule_minutes INTEGER DEFAULT 60,
    last_run_at DATETIME,
    last_success_at DATETIME,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    CHECK (
        (ingestion_mode = 'html_scraper' AND search_url IS NOT NULL)
        OR ingestion_mode IN ('manual_import', 'disabled')
    )
);
```

#### `jobs`

```sql
CREATE TABLE jobs (
    id TEXT PRIMARY KEY,
    source_id TEXT NOT NULL,
    provider TEXT NOT NULL,
    job_url TEXT NOT NULL,
    canonical_url TEXT NOT NULL,
    external_job_id TEXT,
    city_location TEXT,
    job_title TEXT NOT NULL,
    employer_name TEXT,
    employment_type TEXT,
    salary_range TEXT,
    description TEXT,
    posted_at DATETIME,
    first_seen_at DATETIME NOT NULL,
    last_seen_at DATETIME NOT NULL,
    content_hash TEXT,
    raw_payload TEXT,
    is_applied BOOLEAN NOT NULL DEFAULT FALSE,
    applied_at DATETIME,
    is_favorite BOOLEAN NOT NULL DEFAULT FALSE,
    user_comments TEXT,
    status TEXT NOT NULL DEFAULT 'new',
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (source_id) REFERENCES job_sources(id)
);
```

#### Unique constraints

```sql
CREATE UNIQUE INDEX idx_jobs_provider_canonical_url
ON jobs(provider, canonical_url);
```

Optional secondary unique constraint:

```sql
CREATE UNIQUE INDEX idx_jobs_provider_external_id
ON jobs(provider, external_job_id)
WHERE external_job_id IS NOT NULL;
```

#### `ingestion_runs`

```sql
CREATE TABLE ingestion_runs (
    run_id TEXT PRIMARY KEY,
    started_at DATETIME NOT NULL,
    finished_at DATETIME,
    status TEXT NOT NULL,
    new_jobs_count INTEGER NOT NULL DEFAULT 0,
    updated_jobs_count INTEGER NOT NULL DEFAULT 0,
    skipped_jobs_count INTEGER NOT NULL DEFAULT 0,
    error_count INTEGER NOT NULL DEFAULT 0,
    errors TEXT,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);
```

#### `ingestion_run_sources`

Each scheduled or manual full ingestion creates one parent `ingestion_runs` row and one child row per source that was evaluated. Manual single-job imports may create a parent run and a single child row for the selected source.

```sql
CREATE TABLE ingestion_run_sources (
    id TEXT PRIMARY KEY,
    run_id TEXT NOT NULL,
    source_id TEXT NOT NULL,
    provider TEXT NOT NULL,
    ingestion_mode TEXT NOT NULL,
    started_at DATETIME NOT NULL,
    finished_at DATETIME,
    status TEXT NOT NULL,
    new_jobs_count INTEGER NOT NULL DEFAULT 0,
    updated_jobs_count INTEGER NOT NULL DEFAULT 0,
    skipped_jobs_count INTEGER NOT NULL DEFAULT 0,
    error_count INTEGER NOT NULL DEFAULT 0,
    errors TEXT,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    FOREIGN KEY (run_id) REFERENCES ingestion_runs(run_id),
    FOREIGN KEY (source_id) REFERENCES job_sources(id)
);
```

#### Query and filtering indexes

```sql
CREATE INDEX idx_jobs_source_id ON jobs(source_id);
CREATE INDEX idx_jobs_status ON jobs(status);
CREATE INDEX idx_jobs_last_seen_at ON jobs(last_seen_at);
CREATE INDEX idx_jobs_first_seen_at ON jobs(first_seen_at);
CREATE INDEX idx_jobs_is_favorite ON jobs(is_favorite);
CREATE INDEX idx_jobs_is_applied ON jobs(is_applied);
CREATE INDEX idx_jobs_provider ON jobs(provider);
```

SQLite full-text search may be added later for keyword search across `job_title`, `employer_name`, and `description` if basic `LIKE` matching becomes slow.

### 5.2 Deduplication Rules

Primary deduplication rule:

```text
provider + canonical_url
```

Canonical URL normalization must:

1. Lowercase scheme and host.
2. Remove tracking query parameters such as `utm_*`, `tracking`, `ref`, `source`.
3. Remove trailing slashes where safe.
4. Preserve provider-specific job identifiers.
5. Avoid destructive normalization that merges different jobs.

When a job already exists:

1. Update parsed fields if new data is better or changed.
2. Update `last_seen_at`.
3. Recompute `content_hash` from normalized `job_title`, `employer_name`, `city_location`, `description`, and `posted_at` values; use it to detect substantive content changes.
4. Do not overwrite user fields:

   * `is_applied`
   * `applied_at`
   * `is_favorite`
   * `user_comments`
   * `status`, unless status is automatically changed from `new` to `stale` by a stale-job policy.

## 7. Parser/Adapter Design

### 7.1 Parser Interface

Every provider parser must implement a common interface.

```python
class JobSourceParser(Protocol):
    provider: str

    def validate_source(self, source: JobSource) -> None:
        """
        Validate source URL and ensure it is suitable for scraping.
        """

    def discover_job_links(self, source: JobSource, html: str, final_url: str) -> list[DiscoveredJob]:
        """
        Extract job links and metadata from a fetched search-results page.
        """

    def parse_job_detail(self, discovered_job: DiscoveredJob, html: str, final_url: str) -> ParsedJob:
        """
        Extract normalized job fields from a fetched job-detail page.
        """
```

HTTP fetching must be handled by `ingestion/fetcher.py` or an equivalent fetch service, not by parser implementations. Parsers should be deterministic transformations from HTML/content plus metadata to DTOs so they can be tested with saved fixtures.

### 7.2 Data Transfer Objects

```python
class DiscoveredJob(BaseModel):
    provider: str
    source_id: str
    job_url: str
    external_job_id: str | None = None
    title_hint: str | None = None
    employer_hint: str | None = None
    location_hint: str | None = None
    raw_payload: dict | None = None
```

```python
class ParsedJob(BaseModel):
    provider: str
    source_id: str
    job_url: str
    canonical_url: str
    external_job_id: str | None = None
    city_location: str | None = None
    job_title: str
    employer_name: str | None = None
    employment_type: str | None = None
    salary_range: str | None = None
    description: str | None = None
    posted_at: datetime | None = None
    raw_payload: dict | None = None
```

### 7.3 Parser Registry

The backend must not directly import parser classes throughout the codebase. Use a registry.

```python
class ParserRegistry:
    def register(self, parser: JobSourceParser) -> None:
        ...

    def get(self, provider: str) -> JobSourceParser:
        ...
```

Adding a new source should require only:

1. Creating a new parser file.
2. Implementing the parser interface.
3. Registering the parser.
4. Adding source config.

Example:

```text
ingestion/parsers/linkedin.py
ingestion/parsers/seek.py
ingestion/parsers/company_careers.py
```

#### Pagination and discovery behavior

Parser implementations should handle pagination or multi-page search results where supported by the provider.

- `seek.com` listings typically expose page-linked search result URLs or `page` query parameters. A SEEK parser should iterate one or more result pages until it has discovered a stable set of jobs or reached a reasonable page limit.
- `linkedin.com` listings may rely on dynamic content and infinite scroll. A LinkedIn parser should extract the first page of visible search results from HTML or server-rendered page fragments, and may optionally parse additional pagination tokens if available.
- Parsers should avoid aggressive crawling and should only follow provider-supported pagination patterns. The default implementation may limit discovery to the first 2–3 pages or the first N results for reliability.

## 8. Fetcher Policy

The fetcher is responsible for all outbound HTTP behavior. It must provide a single reusable path for scheduled scraping and manual job imports.

Fetcher requirements:

1. Use a clear, configurable `User-Agent` that identifies the local app without impersonating browsers or official clients.
2. Respect provider terms, robots guidance where applicable, and site-specific restrictions. If a provider clearly blocks scraping or requires authentication, mark the source as failed or disabled rather than trying to bypass controls.
3. Apply conservative timeouts, with separate connect and read timeout settings.
4. Retry only transient failures such as network timeouts, `408`, `429`, and `5xx` responses.
5. Use exponential backoff with jitter for retries, and honor `Retry-After` headers when present.
6. Rate-limit requests per provider and per source, with a default delay between requests.
7. Enforce configurable maximums for pages per source, jobs discovered per source, detail pages fetched per run, redirect count, and response body size.
8. Record final URL, status code, response headers needed for debugging, elapsed time, and retry count in structured logs.
9. Classify failures as validation, blocked, rate-limited, transient fetch, permanent fetch, parse, or database errors.
10. Never fetch private, authenticated, or user-specific pages unless the app explicitly adds a secure authenticated-source feature in a future version.

Recommended default limits:

| Setting | Default |
| ------- | ------- |
| `request_timeout_seconds` | 20 |
| `connect_timeout_seconds` | 5 |
| `max_retries` | 2 |
| `default_request_delay_seconds` | 2 |
| `max_pages_per_source` | 3 |
| `max_discovered_jobs_per_source` | 100 |
| `max_detail_fetches_per_source_run` | 50 |
| `max_response_bytes` | 5 MB |

## 9. Ingestion Workflow

### 9.1 Scheduled Ingestion

The backend must support timer-based execution every 30 minutes or every hour.

The scheduler must:

1. Load enabled sources.
2. Skip sources where `ingestion_mode = disabled`.
3. Acquire a lock so only one ingestion run executes at a time.
4. Run each source independently.
5. Respect each source's `schedule_minutes` interval when deciding whether to poll.
6. Log success/failure per source.
7. Continue processing other sources if one source fails.
8. Write ingestion results to the database transactionally.

Scheduling instructions must be provided in `README.md` for WSL/cron setup so the app can be run on a local Windows Subsystem for Linux environment with periodic invocation.

Pseudo-flow:

```text
Start ingestion run
  For each enabled source:
    Validate source
    Get parser from registry
    Fetch search page HTML
    Pass fetched HTML to parser and discover job links
    For each discovered job:
      Canonicalize URL
      Check if job exists
      If missing:
        Fetch job detail HTML
        Pass fetched HTML to parser and parse full job detail
        Insert job
      If existing:
        Optionally refresh details
        Update last_seen_at
    Update source last_run_at and last_success_at
End ingestion run
```

### 9.2 Manual Ingestion Trigger

The UI must include a button:

```text
Update Jobs Now
```

When clicked:

1. Call backend endpoint `POST /api/ingestion/run`.
2. Start ingestion immediately.
3. Prevent duplicate concurrent runs.
4. Show current run status.
5. Show number of new jobs, updated jobs, skipped jobs, and errors.

### 9.3 Manual Job Import

The UI must support a manual "paste job URL" workflow for sources with `ingestion_mode = manual_import`.

Manual import flow:

1. User selects a manual-import source or provider.
2. User pastes a single job detail URL.
3. Backend validates that the source exists, is enabled, and supports `manual_import`.
4. Backend fetches the job detail page through the shared fetcher.
5. Backend passes the fetched HTML to the provider parser.
6. Backend canonicalizes, deduplicates, inserts or updates the job, and preserves user-managed fields.
7. UI displays whether the job was added, updated, skipped, or failed.

Manual import must not require `search_url`.

### 9.4 Ingestion Run Summary

Store or return:

| Field                | Description                                       |
| -------------------- | ------------------------------------------------- |
| `run_id`             | Unique run ID                                     |
| `started_at`         | Start timestamp                                   |
| `finished_at`        | End timestamp                                     |
| `status`             | `running`, `success`, `partial_failure`, `failed` |
| `new_jobs_count`     | Number of inserted jobs                           |
| `updated_jobs_count` | Number of existing jobs updated                   |
| `skipped_jobs_count` | Number skipped                                    |
| `error_count`        | Number of errors                                  |
| `errors`             | Structured error messages                         |

## 12. Job Status Lifecycle

Default status:

```text
new
```

Allowed statuses:

```text
new
viewed
applied
rejected
archived
stale
```

Rules:

1. New jobs start as `new`.
2. Opening a job detail page may optionally change `new` to `viewed`.
3. Marking a job as applied sets status to `applied`.
4. Archiving sets status to `archived`.
5. A job may become `stale` if not seen for a configurable number of days, defaulting to 14 days.
6. The stale threshold should be configurable in settings and stale marking may occur as part of daily ingestion or a dedicated daily cleanup run.
7. User-set statuses should not be overwritten by ingestion except for safe metadata updates.

Status invariants:

1. `status = applied` implies `is_applied = true` and `applied_at` is not null.
2. Setting `is_applied = true` must set `status = applied`.
3. Setting `is_applied = false` must clear `applied_at` unless the request explicitly preserves it, and must move `status` from `applied` to `viewed` if the job has been viewed, otherwise `new`.
4. `status = archived` should not be changed by ingestion or stale cleanup.
5. `status = rejected` should not be changed by ingestion or stale cleanup.
6. Stale cleanup may move `new` or `viewed` jobs to `stale`, but must not move `applied`, `archived`, or `rejected` jobs.

## 13. Error Handling

The app must use structured errors.

Examples:

```json
{
  "error_code": "PARSER_NOT_FOUND",
  "message": "No parser registered for provider seek",
  "source_id": "abc123"
}
```

Common error codes:

```text
SOURCE_DISABLED
INGESTION_MODE_DISABLED
PARSER_NOT_FOUND
SOURCE_VALIDATION_FAILED
FETCH_FAILED
FETCH_BLOCKED
FETCH_RATE_LIMITED
FETCH_TIMEOUT
PARSE_FAILED
DATABASE_WRITE_FAILED
DUPLICATE_JOB
INGESTION_ALREADY_RUNNING
MANUAL_IMPORT_SOURCE_REQUIRED
MANUAL_IMPORT_URL_INVALID
```

Structured error payloads returned by the API should include:

- `error_code`
- `message`
- optional `source_id`
- optional `job_url`
- optional `details` or `metadata`

HTTP response conventions:

- `400` for bad requests or validation failures
- `404` for missing resources
- `409` for duplicate or conflict conditions
- `500` for internal server errors

Parser failures for one job must not stop the whole source run.

Source failure must not stop other sources.
